P13: Time scrubber with rolling 6h aircraft buffer (IndexedDB-backed) and play/pause replay.

Goal: a draggable timeline scrubber along the bottom of the viewport. User drags backwards to replay aircraft positions from the last 6 hours. Live mode is the default; entering replay mode pauses live updates for aircraft only. Play/pause control with 1x and 2x speeds.

SCOPE: Aircraft only. Vessels, satellites, fires, quakes, jamming, airspace, cables — all stay live during replay. Theater fly is unchanged. Vessel/satellite trails (P8) continue rendering live throughout.

DATA BUFFER:

1. New module src/buffer/aircraftBuffer.ts:
   - IndexedDB database name: "architects-eye-buffer"
   - Store name: "aircraft-positions"
   - keyPath: auto-increment id
   - Indexes: timestamp_ms (number), icao24 (string)
   - Schema per record: { id?, icao24, timestamp_ms, lat, lon, alt_baro_ft, ground_speed_kts, track_deg, callsign }
   - On every successful /api/aircraft fetch (the existing 8s poll cycle): for each aircraft in the response, write one record to the buffer with timestamp_ms = Date.now()
   - Throttle/batch: write all aircraft from one fetch in a single IndexedDB transaction (don't open 200 transactions per poll)
   - Eviction: every 5 minutes, run a cleanup that deletes records older than 6 hours (timestamp_ms < Date.now() - 6*3600*1000). Use the timestamp_ms index for efficient range delete.
   - Buffer starts EMPTY on page load. Fills as user watches. This is documented behavior, not a bug. README mentions it.

2. Buffer query API:
   - getPositionsAtTime(timestamp_ms): returns Map<icao24, latestRecordAtOrBefore> — for each unique icao24 in the buffer, return the most recent record with timestamp_ms <= queried time. Returns empty Map if buffer is empty or no records before that time.
   - getBufferRange(): returns { earliest_ms, latest_ms } or null if empty. Used by the scrubber to know its scrub range.

REPLAY STATE (store/index.ts):

3. New store fields:
   - playbackMode: 'live' | 'replay'  (default 'live')
   - replayTimestamp_ms: number | null  (current scrub position when in replay mode)
   - replaySpeed: 1 | 15  (default 1)
   - replayPlaying: boolean  (default false — entering replay starts paused)
   - Actions: enterReplay(timestamp_ms), exitReplay(), setReplayTimestamp(ms), setReplaySpeed(s), togglePlayPause()

4. Replay tick loop:
   - When playbackMode === 'replay' AND replayPlaying === true: a single requestAnimationFrame loop advances replayTimestamp_ms by (deltaFrame_ms * replaySpeed) per tick.
   - When replayTimestamp_ms reaches latest_ms (current real time), automatically exitReplay() — switch back to live mode. UI shows a brief "Returned to live" toast (existing toast system if any; otherwise just console.log and the LIVE pill turns green again).
   - Clock multiplier respects replaySpeed: 1 = real-time, 15 = 15x.

AIRCRAFT LAYER INTEGRATION (src/layers/AircraftLayer.ts):

5. Extend AircraftLayer to subscribe to playbackMode:
   - playbackMode === 'live': existing behavior (poll /api/aircraft, render store.aircraft Map). Unchanged.
   - playbackMode === 'replay': suspend the live aircraft poll (don't write to store.aircraft from network). On every replayTimestamp_ms change, call aircraftBuffer.getPositionsAtTime(replayTimestamp_ms) and render that Map instead. Render path itself doesn't care which Map — same triangle, same trail.
   - When exiting replay → resume live poll, clear any stale replay-rendered entities, store rehydrates from next live poll cycle.

6. Aircraft trails during replay:
   - The P8 PathGraphics 10min cyan trail uses Cesium SampledPositionProperty. During replay, feed it samples from the buffer up to replayTimestamp_ms (window: replayTimestamp_ms - 10min → replayTimestamp_ms). This gives the user a visible motion trail as they scrub.
   - On scrub direction reversal (dragging backward through time), the trail shows the aircraft's recent past relative to the current scrub position — same logic, just queried at the new timestamp.

UI — TIMELINE COMPONENT (components/Timeline.tsx — replaces existing Timeline skeleton from P3):

7. Layout: bottom of viewport, full width, 56px tall, dark background matching header style.

8. Components inside:
   - Left: PLAY/PAUSE button, 32px square, monospace icon. Disabled (greyed) when playbackMode === 'live'.
   - Left-of-center: SPEED toggle "1x" / "2x", monospace pill, 32px tall. Click cycles 1 → 2 → 1. Disabled when 'live'.
   - Center: scrubber bar
     - Track: full width minus left/right control areas, height 4px, dark grey
     - Filled portion: from earliest_ms to current (replayTimestamp_ms when 'replay', else latest_ms). Amber color #fbbf24.
     - Handle: 12px circle, draggable, snaps to current playback position
     - Tick marks: every 1 hour mark within the buffer range, small vertical line
     - Hover label: shows ISO timestamp at hovered scrubber position
   - Right: LIVE pill — green when playbackMode === 'live', dimmed when 'replay'. Click while in replay → exitReplay() → snaps back to live.
   - Far right: UTC clock, existing.

9. Scrubber interaction:
   - Click anywhere on the track → enterReplay(timestamp at click x)
   - Drag the handle → setReplayTimestamp(ms) on every drag event, throttled to 60fps
   - When user drags into the "future" (timestamp >= latest_ms in buffer) → exitReplay()
   - Empty buffer (page just loaded): scrubber is disabled, shows "BUFFER EMPTY · WATCH TO RECORD" placeholder text. After ~30s of buffering, scrubber becomes interactive.

10. Replay-mode visual indicator:
    - When playbackMode === 'replay': add a subtle 1px amber border around the entire viewport, and a "REPLAY · {scrub timestamp UTC}" pill in the top-right of the viewport (next to the LIVE/UTC area). Makes it impossible to confuse replay with live during the Loom recording.

DEFINITION OF DONE:
- Page loads → buffer starts empty → "BUFFER EMPTY · WATCH TO RECORD" shown in scrubber area
- Wait 60 seconds → buffer has 7-8 polls of aircraft data → scrubber becomes interactive
- Click scrubber 30s back → playbackMode flips to 'replay', aircraft jump to their positions 30s ago, trails redraw, LIVE pill dims, REPLAY pill appears, amber viewport border appears
- Click PLAY → replayTimestamp advances at 1x → aircraft visibly move forward in time → trails follow
- Click 2x → playback doubles speed
- Drag scrubber back further → all aircraft snap to that moment, trails recompute
- Click LIVE pill → exits replay, aircraft snap to current live positions, amber border disappears
- Replay reaches latest_ms → auto-exits to live mode, console logs "[REPLAY] returned to live"
- Vessels, satellites, fires, quakes — all keep moving normally during replay (independent of playbackMode)
- Theater fly during replay → camera flies normally, replay continues, no conflict
- Reload page → IndexedDB persists, scrubber immediately shows whatever buffer existed from prior session (within 6h retention)

PERF GUARDRAILS:
- IndexedDB writes batched per-poll (single transaction, not 200)
- Buffer queries indexed on timestamp_ms — at 6h * ~150 aircraft * 8s polls = ~400k records max — cleanup keeps it bounded
- requestAnimationFrame loop only runs while replayPlaying === true
- No setInterval polling on the buffer; it's pull-only (queried on scrub events and in the rAF loop)
