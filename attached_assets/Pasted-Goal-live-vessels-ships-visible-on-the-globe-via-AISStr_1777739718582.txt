Goal: live vessels (ships) visible on the globe via AISStream.io WebSocket.

Data source: AISStream.io — wss://stream.aisstream.io/v0/stream
Authentication: API key in subscription message. Key is in VITE_AISSTREAM_API_KEY (already in workspace + deployment Secrets).

CRITICAL: AISStream is a WebSocket, not HTTP. The browser cannot connect directly because the API key would be exposed AND we want consistent server-mediated traffic. Route the AIS WebSocket through the existing Express server.

Architecture:

Server side:
1. server/src/routes/ais.ts — Express WebSocket route at /ws/vessels:
   - Accepts WebSocket upgrade requests from browser clients
   - Maintains a single persistent connection from server to wss://stream.aisstream.io/v0/stream
   - Aggregates all browser viewport bounding boxes; sends a unioned subscription to AISStream
   - On every browser viewport change, recomputes the union and re-subscribes (debounced 2s)
   - Forwards filtered AISStream messages to relevant browser clients (only ones whose viewport contains the vessel)
   - Auto-reconnects with exponential backoff (1s, 2s, 4s, 8s, 16s capped) on AISStream disconnect
   
2. Subscription message to AISStream:
   {
     "APIKey": process.env.VITE_AISSTREAM_API_KEY,
     "BoundingBoxes": [[[swLat, swLon], [neLat, neLon]]],
     "FilterMessageTypes": ["PositionReport", "ShipStaticData"]
   }

3. Install ws dependency in server package

Client side:

4. src/ws/aisstream-client.ts — class AISStreamClient:
   - Connects to ws://[same-origin]/ws/vessels
   - On connection, sends viewport bbox from store
   - Watches store viewport changes; on change (debounced 1s), sends new bbox to server
   - On PositionReport message: emit event { mmsi, lat, lon, sog, cog, heading, ts }
   - On ShipStaticData message: emit event { mmsi, name, type, callsign, destination, flag }
   - Auto-reconnect with exponential backoff on disconnect
   - Graceful fail: if WebSocket can't connect, hide the vessel toggle in LayerToggles and log a warning, do NOT crash

5. src/layers/VesselLayer.ts:
   - Maintain Map<mmsi, Cesium.Entity> + Map<mmsi, staticData>
   - On PositionReport: update entity position with SampledPositionProperty (in-place update like AircraftLayer, no destroy-recreate)
   - Add future-sample at +30s extrapolated from cog and sog so entity stays visible between updates
   - Set forwardExtrapolationType = HOLD as safety net
   - On ShipStaticData: store metadata, update existing entity's name if visible
   - Use a triangle billboard (programmatic SVG, like aircraft) but in lighter blue (#60a5fa)
   - Heading rotation from cog
   - Stale pruning: remove entities not seen in 5 minutes
   - Click → store.setSelectedEntity({ type: 'vessel', id, data: { name, mmsi, type, flag, sog, cog, destination } })
   - Subscribe to layerVisibility.vessels — toggle entire collection on/off

6. EntityPanel.tsx — extend for type 'vessel':
   - Hero: vessel name (or MMSI if name unknown)
   - MMSI, ship type (decoded from numeric type code — common ones: 70-79 cargo, 80-89 tanker, 60-69 passenger, 30-39 fishing), flag (decoded from MMSI MID prefix), course over ground, speed over ground, destination

7. LayerToggles.tsx — vessels toggle now active/interactive, count updates from store.layerCounts.vessels

8. Wire viewport→bbox conversion: derive [[swLat, swLon], [neLat, neLon]] from viewer.camera.computeViewRectangle(). Send to AISStream client when viewport changes.

Definition of done:
- Pan camera over Mediterranean → see ships moving (lighter blue triangles)
- Click a ship → side panel shows name, MMSI, type, flag, course, speed
- Toggle vessels off → all ships disappear
- AISStream WebSocket reconnects gracefully if dropped

Do not touch jamming, theater system, or time scrubber yet. Vessels only. Do not break aircraft or satellite layers.
